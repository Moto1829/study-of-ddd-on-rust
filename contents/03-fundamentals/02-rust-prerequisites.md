# Rustでドメインを表現するための前提知識

DDDの実装では、Rustの基本機能をどう使うかがそのまま設計に直結します。この章では、後続の章で前提になる観点を整理します。

## struct と enum の使い分け

1つのまとまった概念を表すなら struct、取りうる状態や分岐を閉じたいなら enum が基本です。

たとえば、`TaskTitle` は単一の値を包む struct、`TaskStatus` は `Open` と `Completed` のどちらかを取る enum です。

## newtype pattern

newtype pattern は、既存のプリミティブ型や標準型を独自型で包む方法です。

```rust
pub struct TaskTitle(String);
```

この形にすると、ただの `String` ではなく「妥当性検証済みのタスク名」として扱えます。DDDでValue Objectを表現するときの基本手段になります。

## Result による失敗の表現

ドメインルール違反を panic で扱うのではなく、`Result<T, E>` で返すことで、失敗が設計の一部として見えるようになります。

この本のタスク管理アプリでは、`TaskTitle::new` が `Result<TaskTitle, TaskError>` を返します。これにより、呼び出し側は「空文字が来るかもしれない」という可能性を明示的に扱えます。

## 所有権と責務の境界

Rustの所有権は、ただの文法制約ではありません。どのオブジェクトがどの状態を管理するのかをはっきりさせる圧力として働きます。

DDDではこの制約がむしろ利点になります。

- 誰が更新責任を持つかを曖昧にしにくい
- 不必要な双方向参照を避けやすい
- Aggregate境界を意識しやすい

## トレイトの使いどころ

Rustでは抽象化にトレイトを使えますが、DDDでは導入のタイミングが重要です。特にRepositoryはトレイトで表現しやすい一方、EntityやValue Objectまで何でもトレイトにするとかえって読みにくくなります。

まずは具体型でドメインを固め、差し替えが必要になった境界だけ抽象化する方が、設計の意図が明確になります。

次章からは、実際にValue Objectをどう作るかをサンプルコードに沿って見ていきます。
