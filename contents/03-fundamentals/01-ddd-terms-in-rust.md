# 第3章-1 DDDの基本用語をRust目線で整理する

DDDの議論では多くの用語が登場しますが、重要なのは定義を暗記することではなく、コード上でそれらが何を守る役割なのかを理解することです。

## Entity

Entityは、属性が変わっても同一性で区別されるオブジェクトです。Rustでは、識別子を別型として持たせた struct として表現すると扱いやすくなります。

タスク管理ドメインなら、`Task` は Entity です。タイトルが変わっても、`TaskId` が同じなら同じタスクです。

## Value Object

Value Objectは、値そのものに意味があるオブジェクトです。等価性は識別子ではなく値で判断されます。Rustでは、newtype pattern と相性がよく、妥当性検証の入り口としても使いやすいです。

この本のタスク管理アプリでは、`TaskTitle` が Value Object です。空文字や長すぎる文字列を受け付けない責務を持ちます。

## Aggregate

Aggregateは、不変条件を守るための整合性境界です。すべてのオブジェクトを自由につなげるのではなく、「どの単位で状態を一貫させるか」を決めます。

Rustでは所有権があるため、Aggregate境界を曖昧にした設計は扱いにくくなります。逆に言えば、境界を明確に決めるとコードも素直になります。

## Repository

Repositoryは、Aggregateの保存と取得を抽象化するための仕組みです。ドメイン層から永続化の詳細を隠したいときに導入します。

Rustではトレイトで表現することが多いですが、最初からすべてを抽象化する必要はありません。複数実装が必要になる文脈で導入するのが自然です。

## Domain Service

Domain Serviceは、重要な業務ルールでありながら、特定のEntityやValue Objectに自然に属さない処理を置く場所です。

ただし、単なる手続きの置き場所にすると、ドメインモデルが貧血化します。まずはEntityやValue Objectに置けないかを検討してから導入する方が安全です。

## Application Service

Application Serviceはユースケースの進行役です。入力を受け取り、必要なドメインオブジェクトを組み立て、Repositoryを使って保存し、出力を返します。

ここで重要なのは、業務ルールそのものを大量に持たせないことです。業務ルールはできるだけドメインモデル側に寄せます。

## Domain ServiceとApplication Serviceの違いで迷ったら

DDD学習者が最初につまずきやすいのが、Domain ServiceとApplication Serviceの違いです。見た目はどちらも「何かの処理をするサービス」なので、名前だけでは区別しづらいからです。

判断基準は次の通りです。

- Domain Service: ドメイン判断そのものを提供する
- Application Service: ユースケースの手順を組み立てる

たとえば「完了済みタスクは名前変更できない」は、業務ルールなので本来は `Task` のようなドメイン側に属します。もし複数のドメインオブジェクトをまたいで判断するなら、Domain Serviceが候補になります。

一方で「入力を受け取り、Taskを取得し、renameを呼び、保存して結果を返す」はユースケースの進行です。これはApplication Serviceの責務です。

迷ったときは、次の問いを使うと整理しやすくなります。

- これは業務上の判断か
- それとも処理手順の調停か
- このロジックはUIやAPIが変わっても残るか

UIやAPIが変わっても残る業務判断なら、Application Serviceではなくドメイン側に寄せる余地があります。

## Bounded Context

Bounded Contextは、特定の用語やモデルが有効な境界です。同じ「タスク」という言葉でも、個人タスク管理とプロジェクト進行管理では意味が変わるかもしれません。

小規模な学習用サンプルでは意識しづらい概念ですが、用語の意味が揺れ始めたときに必要になります。重要なのは、「同じ名前の型を使っているか」ではなく、「同じ言葉に同じ業務上の意味を与えられているか」です。

たとえば個人タスク管理の `Task` と、チームの進行管理で使う `Task` は、同じ単語でも別のモデルかもしれません。前者では完了とアーカイブが中心でも、後者では担当者、依存関係、承認フローが重要になることがあります。

この本のタスク管理アプリは1つの Bounded Context に絞っていますが、後半の `Bounded Contextをどう見つけるか` では、どんな兆候が出たら境界を切るべきかをもう少し具体的に扱います。

次章では、これらの概念をRustで表現するために必要な言語機能を整理します。
