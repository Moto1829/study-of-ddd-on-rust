# 第4章-2 Entityを実装する

Entityは、値ではなく同一性で区別されるオブジェクトです。RustでDDDを実装するとき、Entityをただの可変structとして扱うだけでは不十分です。重要なのは、どの状態変更を許し、どの状態変更を禁止するかをメソッドとして閉じ込めることです。

## TaskをEntityとして見る

この本のタスク管理アプリでは `Task` をEntityとして扱っています。タイトルは変更できても、同じ `TaskId` を持つ限り同じタスクです。

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct Task {
    id: TaskId,
    title: TaskTitle,
    status: TaskStatus,
}
```

ここでのポイントは、`id`、`title`、`status` がすべて公開されていないことです。外側から直接書き換えられると、不変条件を守る責任が分散してしまいます。

## 振る舞いとして更新を定義する

Entityの状態変更は、フィールドを書き換える手続きではなく、業務上の振る舞いとして定義します。

このタスク管理アプリでは、`Task::rename` と `Task::complete` がそれにあたります。

- `rename` は完了済みタスクには使えない
- `complete` は二重完了を許さない

このように、Entityのメソッドが「何を変更するか」だけでなく「どんな条件で許されるか」も引き受けることで、ルールを1箇所に閉じ込められます。

## RustでEntityを実装するときの利点

Rustでは、可変参照が必要な操作は `&mut self` を明示的に要求します。そのため、どの処理が状態変更を起こすかがシグネチャから読み取りやすくなります。

また、Value Objectを引数に取るようにしておけば、Entity側は「妥当な値だけが渡される」という前提でロジックを組み立てられます。

## 何をEntityメソッドに置くべきか

Entityメソッドに置くべきなのは、そのEntityの整合性を守るために必要なルールです。逆に、外部システムアクセスや複数Aggregateをまたぐ調整のような処理は、別の層に出した方が見通しがよくなります。

実務では「とりあえずsetterを公開する」設計に流れやすいですが、DDDではその方針は避けた方が安全です。setterは変更の自由度を上げますが、同時にルールの責任を曖昧にします。

## よくある誤解

- Entityはデータだけにして、ロジックは全部Serviceへ出すべきだと思いがち
- setterを細かく生やした方が使いやすいと考えがち
- Entityにロジックを入れると責務過多になると誤解しがち

実際には、Entity自身の状態変更条件をEntityが持たないと、不変条件が外へ漏れます。複雑さを減らすためにロジックを出すのではなく、責務の所属先が本当にEntity外かを見極めることが重要です。
