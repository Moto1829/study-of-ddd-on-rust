# 永続化とテストを実装する

DDDを学ぶとき、永続化やテストは後回しにされがちですが、実際にはここが設計の妥当性を確認する重要な場所です。モデルがきれいに見えても、保存やテストで破綻するなら境界設計を見直す必要があります。

## インメモリRepositoryで検証する

このサンプルでは `InMemoryTaskRepository` を使っています。内部実装は単純な `HashMap` ですが、学習用としては十分です。

- Repository境界が本当に必要か確認できる
- Application Serviceを外部依存なしでテストできる
- ドメインルールを小さく繰り返し検証できる

## どこをテストすべきか

最低限、次の2層を分けてテストすると理解しやすくなります。

### ドメイン層のテスト

- Value Objectの妥当性検証
- Entityの状態遷移
- 不変条件違反の確認

### アプリケーション層のテスト

- 新規作成時の重複チェック
- 未存在タスクの取得失敗
- 完了処理のユースケース成立
- アーカイブ処理のユースケース成立
- 一覧取得の絞り込みとページング

この分け方にしておくと、ルール破壊がどの層で起きたかを追いやすくなります。

## DB実装へ進むときの注意

将来的にSQLiteやPostgreSQLを導入する場合でも、Domain Modelを直接ORMモデルにしてしまわない方が安全です。永続化都合のフィールドやライブラリ依存がドメイン層へ侵入すると、学習目的でせっかく整理した責務が崩れます。

今回のサンプルでは、その橋渡しの最小例として `TaskRow` を追加しています。これはDBレコード相当の構造体であり、`TryFrom<TaskRow> for Task` を通してドメインモデルへ変換します。

最初のうちは、次の方針を守るだけでも十分です。

- ドメインモデルは domain 層に閉じる
- Repository実装は infrastructure 層に置く
- ユースケースは application 層から組み立てる

この分離が維持できていれば、技術選定を変えてもドメインの説明を保ちやすくなります。
