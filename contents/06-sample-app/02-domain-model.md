# ドメインモデルを実装する

サンプルアプリケーションのドメインモデルは、Value ObjectとEntityを中心に構成しています。ここでは「何をどの型で表すか」よりも、「どのルールをどこに置くか」を見ることが重要です。

## Value Object

`TaskId` と `TaskTitle` はどちらもValue Objectです。

- `TaskId` は空文字を拒否する
- `TaskTitle` は空文字と長すぎる文字列を拒否する

これにより、Entityを組み立てる前の段階で不正値を遮断できます。

## Entity

`Task` はEntityです。識別子、タイトル、状態を持ち、タイトル変更や完了といった振る舞いを提供します。

ここでの要点は、`Task` が自分自身の整合性を守る責務を持つことです。状態変更が `rename` や `complete` を通ることで、完了済みタスクの再編集のような不正操作を防げます。

## エラー表現

このサンプルでは `TaskError` を列挙型で定義しています。これは単なる例外メッセージではなく、どのルールに違反したのかを型で区別するためです。

学習用途では、ドメインエラーを列挙型で明示するだけでも、設計の意図がかなり見えやすくなります。

## Aggregate RootとしてのTask

現状のモデルでは、`Task` 自身がAggregate Rootです。Repositoryは `Task` を単位として保存し、Application Serviceも `Task` を読み出して更新し、再保存します。

この形にしておくと、どのルールがどこで守られているかが追いやすく、DDDの導入段階には適しています。

また、拡張後のサンプルでは `TaskStatus::Archived` を追加し、「削除」を単なる物理削除ではなく状態遷移として扱います。こうした追加ルールも `Task` 自身が受け持つことで、UIやDB都合ではなくドメイン都合で振る舞いを整理しやすくなります。

## モデルを読み解く順番

このリポジトリのコードを読むときは、次の順番が理解しやすいです。

1. `TaskId` と `TaskTitle` を見る
2. `TaskStatus` と `Task` を見る
3. `TaskRepository` を見る
4. `TaskApplicationService` を見る
5. `TaskSummaryReader` と `TaskRow` を見る

この順番で読むと、下位のドメイン制約が上位のユースケースへどう持ち上がり、さらに一覧取得や永続化モデル変換へどうつながるかがわかります。
