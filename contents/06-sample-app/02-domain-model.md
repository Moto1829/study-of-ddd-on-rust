# ドメインモデルを実装する

タスク管理アプリのドメインモデルは、Value ObjectとEntityを中心に構成しています。ここでは「何をどの型で表すか」よりも、「どのルールをどこに置くか」を見ることが重要です。

## Value Object

`TaskId` と `TaskTitle` はどちらもValue Objectです。

- `TaskId` は空文字を拒否する
- `TaskTitle` は空文字と長すぎる文字列を拒否する

これにより、Entityを組み立てる前の段階で不正値を遮断できます。

## Entity

`Task` はEntityです。識別子、タイトル、状態を持ち、タイトル変更や完了といった振る舞いを提供します。

ここでの要点は、`Task` が自分自身の整合性を守る責務を持つことです。状態変更が `rename` や `complete` を通ることで、完了済みタスクの再編集のような不正操作を防げます。

## エラー表現

このタスク管理アプリでは `TaskError` を列挙型で定義しています。これは単なる例外メッセージではなく、どのルールに違反したのかを型で区別するためです。

学習用途では、ドメインエラーを列挙型で明示するだけでも、設計の意図がかなり見えやすくなります。

## Aggregate RootとしてのTask

現状のモデルでは、`Task` 自身がAggregate Rootです。Repositoryは `Task` を単位として保存し、Application Serviceも `Task` を読み出して更新し、再保存します。

この形にしておくと、どのルールがどこで守られているかが追いやすく、DDDの導入段階には適しています。後続の章で状態遷移を増やす場合も、追加ルールを `Task` 自身が受け持つ形を保ちやすくなります。

## モデルを読み解く順番

このタスク管理アプリを読み解くときは、次の順番が理解しやすいです。

1. 入力値を守る `TaskId` と `TaskTitle` を見る
2. 状態遷移を持つ `TaskStatus` と `Task` を見る
3. 保存境界である `TaskRepository` を見る
4. ユースケースを組み立てる `TaskApplicationService` を見る

この順番で読むと、下位のドメイン制約が上位のユースケースへどう持ち上がるかがわかります。読み取りモデルや永続化モデルへの広がりは、後続の章で段階的に確認します。

次章では、このドメインモデルをユースケースとしてどう使うかを見ます。
