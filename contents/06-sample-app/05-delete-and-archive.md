# 削除ではなく状態遷移として表現する

業務システムでは「削除」は非常によく出てきます。しかし、ドメインの観点では本当に物理削除してよいとは限りません。むしろ多くのケースで、削除は状態遷移として表現した方が安全です。

## タスク管理アプリで Archive を追加する理由

タスクを完全に消してしまうと、履歴確認、監査、誤操作の復旧などが難しくなります。そのため、このタスク管理アプリでは `TaskStatus::Archived` を追加し、削除ではなくアーカイブという状態遷移で扱う形へ拡張します。

```rust
pub enum TaskStatus {
    Open,
    Completed,
    Archived,
}
```

これにより、「もう通常操作の対象ではないが、記録としては残したい」という実務上よくある要件を表現できます。

## アーカイブを振る舞いとして持たせる

アーカイブは単なるフラグ更新ではなく、Entityの振る舞いとして表現した方がよいです。

```rust
impl Task {
    pub fn archive(&mut self) -> Result<(), TaskError> {
        if self.status == TaskStatus::Archived {
            return Err(TaskError::TaskAlreadyArchived);
        }

        self.status = TaskStatus::Archived;
        Ok(())
    }
}
```

この形にしておくと、「アーカイブ済みを再アーカイブできない」「アーカイブ後に編集できない」といったルールをドメインモデルに閉じ込めやすくなります。

## Application Service ではどう扱うか

外部からは `archive_task` のようなユースケースとして公開します。ここでもApplication Serviceは手順を調停するだけで、アーカイブ可能かどうかの判断そのものは `Task` に委譲します。

## 実務で考えるべき点

- 論理削除後の一覧表示からどう除外するか
- 管理者だけはアーカイブ済みも見えるようにするか
- 完全削除が必要なとき、別ユースケースに切り出すか

このように、削除を状態遷移で表すと、ドメインルールと監査要件を両立しやすくなります。
