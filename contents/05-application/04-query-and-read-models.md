# 第5章-4 一覧取得と検索をどう設計するか

登録や更新のユースケースはDDDの入門で扱われやすい一方、一覧取得や検索は後回しにされがちです。しかし実務では、むしろ一覧画面や検索APIの方が頻繁に必要になります。

この章は、一覧取得と検索を概念レベルで整理する章です。後続の第6章では、ここで述べた考え方を具体的な型と実装へ対応付けます。

## 一覧取得は常にAggregateを返すべきか

結論から言うと、常にそうとは限りません。書き込みのために厳密な整合性を持つAggregateと、画面表示のための一覧データは要求が違うからです。

たとえば、タスク一覧画面では次の情報だけ欲しいことがあります。

- ID
- タイトル
- 状態
- 期限や担当者の要約

このとき、毎回完全なAggregateを読み出すより、一覧表示専用の読み取りモデルを返した方が責務が明確になります。

## タスク管理アプリでの分け方

この本のタスク管理アプリでは、書き込み側の `TaskRepository` とは別に、読み取り側として `TaskSummaryReader` を追加しています。これはCQRSを全面採用するというより、「一覧取得は別モデルで扱ってよい」という実務的な整理です。

```rust
pub struct FindTasksQuery {
    pub status: Option<TaskStatusDto>,
    pub page: u64,
    pub per_page: u64,
}

pub struct TaskSummary {
    pub id: String,
    pub title: String,
    pub status: TaskStatusDto,
}
```

ここで重要なのは、`TaskSummary` がドメインモデルそのものではないことです。これは表示や一覧取得のための読み取り専用データです。

言い換えると、この章で押さえたいのは「更新モデルと読み取りモデルは同じでなくてよい」という原則です。実際にどの型をどこへ置くかは、第6章で改めて確認します。

## 検索条件をどこへ置くか

一覧取得や検索では、次のような条件がよく出ます。

- ステータスで絞り込む
- ページングする
- 並び替える
- フリーワード検索する

こうした条件は、書き込み用Repositoryへ無理に押し込むより、Query用のモデルとして明示した方が読みやすくなります。書き込みと読み取りの関心事を混ぜすぎないことが大切です。

## よくある誤解

- 一覧取得でも必ずAggregateを返すべきだと思いがち
- Queryを分けると大げさなCQRSになると感じやすい
- 一覧画面用DTOを作るのは冗長に見えやすい

実際には、更新責務と表示責務は別物です。特に実務では一覧の要求が肥大化しやすいため、早い段階で読み取りモデルを分ける発想を知っておくと設計しやすくなります。
