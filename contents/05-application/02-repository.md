# Repositoryをトレイトで定義する

Repositoryは、Aggregateの保存と取得を抽象化するための境界です。Rustでは、これをトレイトで表現すると依存方向を明確にしやすくなります。

## タスク管理アプリのRepository

この本のタスク管理アプリでは、`TaskRepository` を次のような最小インターフェースで定義しています。

```rust
pub trait TaskRepository {
    fn save(&mut self, task: Task);
    fn find_by_id(&self, task_id: &TaskId) -> Option<Task>;
}
```

ここで大切なのは、Repositoryが永続化技術そのものを表すのではなく、「ドメイン側が必要とする操作」を表している点です。

## なぜ最小限に保つのか

初期段階で検索条件や汎用クエリを大量に持たせると、Repositoryがデータアクセスの便利箱になりやすくなります。まずは、ユースケースが本当に必要とする操作だけを定義した方が設計意図が保ちやすいです。

このタスク管理アプリでは、作成・更新・取得に必要な `save` と `find_by_id` だけに絞っています。

## Rustでトレイトにする意味

トレイト化の利点は、Application Service が永続化の詳細を知らなくて済むことです。

- インメモリ実装でテストできる
- 後からDB実装へ差し替えられる
- ユースケースの責務が明確になる

一方で、何でもトレイトにする必要はありません。Value ObjectやEntityまで抽象化すると、かえって設計の輪郭がぼやけます。Repositoryのように「境界として差し替えが必要」な場所に限定して使うのが妥当です。

## よくある誤解

- DDDならとにかくRepositoryを増やすべきだと思いがち
- Repositoryを汎用DAOのようにして何でも載せたくなりがち
- 最初から複雑な検索APIまで設計しないといけないと感じやすい

Repositoryは便利箱ではなく、Aggregateの保存と取得の境界です。学習段階では、まずユースケースに必要な最小操作だけを定義し、それ以上は必要になってから足す方が設計意図を保ちやすいです。

## 実務的なアドバイス: find_by_filter は危険か

実務では、Repositoryに `find_by_filter` のような汎用メソッドを作りたくなることがあります。柔軟に見えますが、DDDの観点では注意が必要です。

問題は、重要なドメインルールがApplication Service側へ漏れやすくなることです。たとえば「退会済みユーザーを除外する」というルールが本来は常に必要なのに、呼び出し側が filter 条件へその指定を入れ忘れると、単なる検索漏れでは済まず、情報漏洩のようなバグにつながることがあります。

つまり、汎用 filter 方式は便利である代わりに、「どの条件が業務上必須なのか」という知識をアプリケーション層へ持ち出しやすくします。これは責務分担としてあまりよくありません。

DDDでは、こうした重要な条件を表すときほど、意味のある名前を持ったメソッドをRepositoryに定義する方が安全です。

```text
find_by_filter(Filter { status: "Active" })
find_active_users()
```

前者は、App層が「有効ユーザーの定義」を知りすぎています。後者なら、App層は「有効なユーザーを取得したい」とだけ表明し、具体的な判定条件はRepository実装側に閉じ込められます。

もちろん、すべての検索を個別メソッドに分解すればよいわけではありません。ただし、ドメイン上重要な意味を持つ条件については、汎用 filter に押し込まず、明示的な名前を付けた操作として表現する方が、設計意図も安全性も保ちやすくなります。

## 返り値に Option を使う理由

`find_by_id` は未存在を自然に表したいので `Option<Task>` を返しています。これは「見つからないことが通常の分岐である」とみなしているからです。DB障害やI/O失敗を扱う段階になれば、`Result<Option<T>, E>` の形も検討できます。
