# Application Serviceでユースケースを表現する

Application Serviceは、ユーザーの操作やAPIリクエストに対応するユースケースを表現する層です。ドメインモデルを直接外側にさらすのではなく、ユースケース単位で処理をまとめることで、責務を整理できます。

## 何を担当するのか

このサンプルの `TaskApplicationService` は、次のような処理を担当します。

- タスクを新規作成する
- タスクを取得する
- タスク名を変更する
- タスクを完了する

重要なのは、これらのメソッドが業務ルールそのものを大量に持たないことです。業務ルールは `Task` や `TaskTitle` に委譲し、Application Service はユースケースの順番を組み立てます。

## 処理の流れ

たとえばタスク完了ユースケースは、おおむね次の手順になります。

1. 入力されたIDを `TaskId` に変換する
2. Repositoryから `Task` を取得する
3. `Task::complete` を呼ぶ
4. 更新済みの `Task` を保存する
5. 出力用のDTOに変換して返す

この流れはドメインモデルの代わりではなく、ドメインモデルを使うための調停役です。

## DTOを返す理由

Application Serviceの出力には `TaskOutput` を使っています。これは次の理由からです。

- 外部に公開したい形を明示できる
- ドメインモデルの内部構造をそのまま漏らさずに済む
- 将来的にAPIレスポンスや画面表示用の形式へ変換しやすい

小規模なサンプルでは直接 `Task` を返しても動きますが、層の責務を見失いやすくなるので、学習段階からDTOに分けておく方が理解しやすいです。

## よくある誤解

- Application Serviceに処理が集まるなら、業務ルールも全部ここに置けばよいと思いがち
- Domain Serviceとの違いが見えず、両方を同じ手続き置き場として使いがち
- DTOを作るのは過剰設計だと感じやすい

Application Serviceはあくまでユースケースの進行役です。業務ルールの本体をここに集め始めると、ドメインモデルが薄くなり、変更のたびにユースケース層が肥大化します。

## エラーの扱い

`TaskApplicationError` は、ドメインエラーとユースケース上の失敗を分ける役目を持ちます。たとえば、重複IDや未存在タスクはApplication Serviceの文脈で解釈した方が読みやすくなります。

これは例外を増やしたいからではなく、「どの層で起きた失敗か」を見通しやすくするためです。

## トランザクションとの関係

実務では、多くのApplication Serviceがトランザクション境界の中心になります。どこからどこまでを1つのユースケースとして成功させるか、という責務はドメインモデルよりもApplication Serviceに近いからです。
