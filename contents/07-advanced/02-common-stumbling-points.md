# DDD学習者がよくつまずく点

DDDの学習で難しいのは、用語を覚えることよりも、責務の置き場所を見誤らないことです。この章では、初学者が特によく引っかかる点を短く整理します。

## Application ServiceとDomain Serviceはどう違うのか

これはもっとも頻出する混乱です。結論から言えば、違いは「業務判断を持つか」「手順を調停するか」にあります。

- Domain Serviceは、ドメイン上の判断を提供する
- Application Serviceは、ユースケースの進行を担う

たとえば「あるユーザーは未完了タスクを5件までしか持てない」というルールは業務判断です。これはEntity単体に置きにくければDomain Serviceの候補です。

一方で「タスク作成リクエストを受け取り、重複確認し、Taskを作り、保存し、レスポンスを返す」はユースケースの流れなのでApplication Serviceです。

区別に迷うなら、次の見方が有効です。

- そのロジックは業務ルールとして説明できるか
- それとも処理の順番としてしか説明できないか

前者ならDomain Service側、後者ならApplication Service側に寄せるのが基本です。

## Entityにどこまでロジックを持たせるべきか

「ロジックを持つと複雑になるから、Entityはデータだけにした方がよい」と考えられがちですが、DDDでは逆です。Entityが自分の整合性を守れないと、ルールが外へ漏れます。

少なくとも、そのEntity自身の状態変更条件はEntityに持たせる方が安全です。

- タイトル変更の可否
- 状態遷移の可否
- 内部整合性の維持

これらを外のServiceへ出すと、同じルールを複数箇所で守る必要が出てきます。

## Value Objectと単なる型ラッパーの違いは何か

newtypeで包んだだけでは、まだValue Objectとして不十分なことがあります。Value Objectとして意味があるのは、ドメイン上の意味と制約を持っているときです。

たとえば `TaskTitle` は、文字列を包むだけでなく、空文字不可や文字数制限を引き受けています。この責務があるから、Value Objectとして価値があります。

逆に、意味も制約もない単なるラッパーを大量に作ると、型は増えるのに設計は改善しません。

## Repositoryはどのタイミングで導入すべきか

Repositoryは便利ですが、最初から何でもRepositoryにすると設計がぼやけます。導入の理由は「永続化詳細を隠したい」「取得と保存の境界を明確にしたい」であって、「DDDではRepositoryが必要だから」ではありません。

学習用や小規模実装では、次の基準で十分です。

- Aggregate単位で保存・取得したい
- Application Serviceから永続化詳細を消したい
- インメモリ実装とDB実装を差し替えたい

この条件が揃っていれば、Repositoryを導入する意味があります。

## ドメインモデルとDBモデルは分けるべきか

原則としては分けた方が安全です。理由は、DB都合の表現とドメイン都合の表現が一致しないことが多いからです。

- DBではNULL許容だが、ドメインでは必須
- DBでは複数カラムだが、ドメインでは1つのValue Object
- DBでは正規化の都合があるが、ドメインでは振る舞い中心で考えたい

学習用サンプルでは差が小さく見えるかもしれませんが、実務で規模が上がるほど差は大きくなります。

## DDDは最初から全部導入しないと意味がないのか

そうではありません。むしろ最初から全部入れると、学習コストだけが上がりやすいです。最初は次の3点で十分です。

- 不変条件をValue ObjectやEntityに閉じ込める
- ユースケースをApplication Serviceとして分ける
- 永続化をRepository境界の外へ出す

これだけでも、単なるCRUD実装との差はかなりはっきり出ます。

## Rustでは所有権が邪魔にならないか

最初は邪魔に感じやすいですが、多くの場合は設計の曖昧さを表面化させているだけです。共有可変状態が必要に見えるときは、本当にその参照関係が必要かを疑った方がよいです。

RustでDDDを進めるときは、所有権制約を回避するよりも、Aggregate境界や責務分担を見直す方が良い結果になりやすいです。

## 学習を進める順番はどうするべきか

順番を欲張らない方が定着します。おすすめは次の順です。

1. Value Objectで不変条件を表現する
2. Entityに状態変更のルールを持たせる
3. Application Serviceでユースケースを組み立てる
4. Repositoryで永続化境界を切る
5. 必要になったらDomain Serviceや発展パターンを考える

この順番なら、用語に引きずられず、責務の置き場所から理解できます。
