# 第7章-9 Bounded Contextをどう見つけるか

DDDを学び始めた段階では、Entity や Value Object までは理解しやすくても、Bounded Context は急に抽象度が上がったように感じやすい概念です。しかし実務では、モデルを壊す原因の多くが「同じ言葉を別の意味で使い始めること」にあります。Bounded Context は、その混線を防ぐための境界です。

## Bounded Contextは何のためにあるのか

Bounded Contextは、用語、モデル、ルールが一貫して意味を持てる範囲を定めるためのものです。

重要なのは、単にシステムを分割することではありません。同じ言葉でも、業務の見方が変われば意味が変わります。その違いを無理に1つのモデルへ押し込むと、コードも会話も曖昧になります。

たとえば `Task` という言葉でも、次のように意味が変わります。

- 個人タスク管理: 自分用の作業メモ。完了・未完了・アーカイブが中心
- プロジェクト進行管理: 担当者、依存関係、工数、承認フローを持つ作業項目
- カスタマーサポート: 問い合わせ対応単位。SLA や優先度、エスカレーションが重要

どれも表面上は `Task` ですが、守るべきルールも必要な属性も違います。これらを1つの `Task` 型で表そうとすると、モデルはすぐに歪みます。

## いまのサンプルでは1つのBounded Contextに留めている

この本のタスク管理アプリは、個人向けタスク管理という1つのBounded Contextに意図的に絞っています。そのため、`Task`、`TaskTitle`、`TaskStatus` といった用語は、すべて同じ意味で扱えます。

この制約があるからこそ、学習の初期段階では次の点に集中できます。

- Value Objectで何を守るか
- Entityにどこまで振る舞いを持たせるか
- Repositoryをどこで切るか
- 読み取りモデルをいつ分けるか

最初から複数コンテキストを入れないのは、重要な概念を省略したいからではありません。まず1つのコンテキストでモデルを安定させてから、言葉の意味が揺れる場面を扱った方が理解しやすいからです。

## 分けるべき兆候

Bounded Context を検討した方がよい兆候は、次のようなものです。

- 同じ用語に対して、部署や画面ごとに説明が変わる
- ある画面では必須の属性が、別の画面では不要どころか邪魔になる
- 片方の都合で追加したルールが、もう片方では不自然に見える
- 会話の中で「この Task はどっちの意味か」を毎回補足する必要がある
- 1つのモデルに例外分岐が増え続ける

典型例は、営業、会計、物流で同じ `Order` を使っているつもりなのに、実際には見ている対象が違うケースです。こういうときに必要なのは、巨大な共通モデルではなく、意味の境界を認めることです。

## 分けない方がよい場面

逆に、次の段階では無理に Bounded Context を切らない方が安全です。

- まだ用語の揺れが起きていない
- 単にモジュール数が増えただけで、意味の違いはない
- 画面が複数あっても、同じルールを同じ言葉で説明できる
- チームがまず単一モデルで責務分担を学んでいる段階にある

Bounded Context は、分割のための分割ではありません。意味の違いが現れていないのに先回りして切ると、今度は境界の説明コストだけが増えます。

## Rustでどう表現すると自然か

Rustでは、Bounded Context を最初からクレート分割で表現する必要はありません。まずは名前と依存方向で境界を表現する方が現実的です。

```text
src/
  personal_task/
    domain/
    application/
  project_workflow/
    domain/
    application/
  shared/
```

このように、コンテキストごとに言葉の有効範囲を分けるだけでも効果があります。ここで重要なのは `shared` を安易に増やしすぎないことです。共有できるのは、本当に意味が共通なものだけに絞るべきです。

もし境界が十分に固まり、ビルド単位でも強制したくなったら、その時点で Cargo workspace や複数クレートを検討すれば十分です。

## Context Mapをどこまで意識すべきか

複数の Bounded Context が出てきたら、次に必要になるのが関係の整理です。どちらが上流で、どちらが翻訳するのかを曖昧にすると、境界を切っても結局モデルが混ざります。

学習段階では、まず次の問いだけでも十分です。

- どのコンテキストの言葉を正とするか
- 別コンテキストへ渡すときに翻訳が必要か
- 同じ名前でも別物として扱うべきか

ここまで意識できると、Bounded Context は単なる用語説明ではなく、モデルを守る実践的な道具として見えてきます。

## まとめ

Bounded Context は、大規模システム専用の話ではありません。言葉の意味がずれ始めたときに、モデルを壊さずに前へ進むための境界です。

この本のタスク管理アプリは1つのコンテキストに留めていますが、それは Bounded Context を無視しているのではなく、まず単一コンテキストの設計を確実に理解するためです。複数コンテキストへ進むのは、その次の段階です。
